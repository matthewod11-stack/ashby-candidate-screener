# Ashby Screener — Project Context

Python pipeline that pulls candidates from Ashby, ranks them with tiered LLM evaluation across one-or-many roles, renders an HTML report per role, and posts a weekly summary to Slack.

## Stack

- Python **3.14.4** (in `.venv/`; verify before assuming compatibility on a new machine)
- `anthropic` SDK — Sonnet for triage (Stage 2), Opus for deep eval + synthesis (Stages 3–4)
- `httpx` — Ashby REST API client
- `jinja2` — HTML report rendering
- `pyyaml` — config; `python-dotenv` — secrets

Required env vars (in `.env`): `ASHBY_API_KEY`, `ANTHROPIC_API_KEY`.

## Quick Start

```bash
# First-time setup
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # then fill in ASHBY_API_KEY and ANTHROPIC_API_KEY

# Run the pipeline
python run.py                          # default role: staff-backend-engineer
python run.py --role bdr               # specific role
python run.py --all                    # every role in config/roles/
python run.py --role bdr --resume      # skip stages whose JSON output exists
python run.py --demo                   # zero-key demo: sample report from synthetic data
python run.py --source local --input-dir ./candidates/   # run on a local résumé folder (no ATS)
```

No test suite is configured.

## Structure

```
run.py                          # Orchestrator — argparse → load_role_config → run_role(s)
models.py                       # Dataclasses + load_json/save_json helpers
state.py                        # Per-role state.json: dedup + cached evals
usage.py                        # UsageTracker + PRICES; per-run token + cost ledger
stages/
  _shared.py                    # Résumé→content-block loader + shortlist_rows (shared helpers)
  sources.py                    # Stage 1 source dispatcher (ashby | local)
  fetch.py                      # Stage 1 (ashby): Ashby API → raw candidates JSON (parallel)
  local_source.py               # Stage 1 (local): folder of résumés + candidates.csv → JSON
  triage.py                     # Stage 2: Sonnet pass/fail at score threshold
  deep_eval.py                  # Stage 3: Opus scores on configured dimensions
  synthesize.py                 # Stage 4: Opus produces tiering + interview questions
  report.py                     # Stage 5: Jinja2 → HTML (per role + combined w/ priority view)
  export.py                     # Shortlist export (CSV + Markdown) for hiring managers
  demo.py                       # Zero-key demo: mock synthesis → report
  slack_summary.py              # Post-pipeline Slack message builder
config/
  roles/                        # One YAML + one Markdown JD per role (see Roles below)
  criteria.yaml                 # Legacy single-role config (used by report.py's standalone demo)
  jd.md                         # Legacy single-role JD (fallback if a role omits jd_file)
  exemplars.yaml                # Few-shot examples for LLM prompts (optional)
  scheduled-task-prompt.md      # Prompt used by the weekly scheduled task
templates/report.html           # Jinja2 template (single + combined report shapes)
data/                           # (gitignored) all runtime output
  pdfs/                         # Shared resume cache (across all roles)
  roles/<slug>/
    results/                    # Stage output JSON files
    reports/                    # HTML reports
    state.json                  # Per-role candidate registry + eval cache
```

> These are **baseline templates** — generic example roles so you can run the pipeline out of the box. Replace the company details, criteria, and scoring dimensions in each YAML (and the matching `.md` job description) with your own before using for real hiring.

Each role's YAML controls: scoring dimensions, model IDs, triage threshold, report `min_score` filter, and `exclude_tiers` (e.g., `"Tier 4 – Pass"` — note the en-dash). The matching `.md` file is the full job description fed into prompts.

## Stage Output Contract

Each stage reads the previous stage's JSON and writes its own. All paths are relative to `data/roles/<slug>/`:

| Stage | Output | Contains |
|---|---|---|
| 1 | `results/stage1_candidates.json` | All raw candidates from Ashby |
| 1.5 | `results/stage1_new_candidates.json` | New-only after dedup against `state.json` |
| 2 | `results/stage2_triage.json` | `{passed, score, reasoning}` per candidate |
| 3 | `results/stage3_deep_evals.json` | Full scored evals for triage survivors only |
| 3.5 | `results/stage3_merged_evals.json` | New evals + cached evals (returning candidates) |
| 4 | `results/stage4_synthesis.json` | Final tiering + interview questions |
| 5 | `reports/report_<date>.html` | Per-role report |
| post | (Slack message text) | Generated by `slack_summary.py`, non-fatal on failure |

State (`state.json`) is keyed by lowercased email; stores `cached_eval` + `cached_synthesis` so returning candidates skip Stages 2 and 3 on subsequent runs.

## Multi-Role Mode

With `--all`, `run.py` iterates every slug from `config/roles/*.yaml` sequentially, then renders a combined report via `generate_combined_report` (`stages/report.py`). The combined report leads with a consolidated **"who to meet this week"** priority list — top-two-tier candidates across all roles, score-sorted — above the per-role tabs, and writes a combined shortlist (`data/reports/shortlist_combined.csv|.md`). Each single-role run also writes a per-role `shortlist.csv|.md`.

## Conventions

- **Per-role config is the unit of customization.** To target a new role, add `config/roles/<slug>.yaml` + `config/roles/<slug>.md`. No code changes required for a new role.
- **Model IDs live in role YAML, never in stage modules.** Each role can use different models if you want to experiment.
- **Stage modules fail fast.** The orchestrator catches and aborts on any exception with a clear stage label.
- **Score thresholds, not count caps.** Use `report.min_score` and `exclude_tiers` to control output volume — never a hard `top_n` cutoff.
- **Tier strings are exact.** `"Tier 4 – Pass"` uses an en-dash (`–`), not a hyphen. Match exactly when filtering.
- **Resumability is per-stage.** `--resume` checks for the existence of each stage's JSON output and skips if present. Delete a stage's JSON to force a re-run. Stage 4 additionally caches the raw Opus response at `results/stage4_synthesis_raw.txt` so a parse-time crash can re-parse without a fresh API call.
- **Per-call cost is logged.** Each Anthropic API call records token usage via `UsageTracker` (in `usage.py`). End-of-run summary prints per-stage and total cost; full history appended to `data/roles/<slug>/usage.json`. Pricing constants in `usage.py:PRICES`.
- **Prompt caching is on by default.** Triage and deep-eval system prompts are sent as cache_control blocks; cache reads are 10% of input rate. Concurrent calls within the 5-min cache TTL get the discount automatically.
- **State persists across runs.** Cached evals carry forward — a candidate evaluated in run N appears in run N+1's report without re-running Stage 2 or 3 (unless you delete `state.json`).

## Anti-Patterns

- Do NOT hardcode a specific company or client name in prompt templates, report rendering, or scoring logic — keep the core company- and ATS-agnostic.
- Do NOT add hard count cutoffs (top 20, top 10) — use score thresholds and tier filters.
- Do NOT build a web server or database — this is a CLI pipeline with local file storage.
- Do NOT add the Slack SDK or a bot token — delivery is via Claude Desktop MCP connector reading `slack_summary.py` output.
- Do NOT add dependencies beyond what's in `requirements.txt` without justification — minimal surface area is a feature.
- Do NOT cross role boundaries in `data/roles/<slug>/` — each role's state and output are isolated.
